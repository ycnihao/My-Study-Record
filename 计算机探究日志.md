            ### 🛣️ WSL 代理：一场从“暴力破解”到“机制通透”的排错之旅

            #### 1. 第一阶段：朴素的直觉（失败）
            *   **目标**：让 WSL 能上网。
            *   **尝试**：直接使用 `127.0.0.1` 或手动查到的 Windows IP。
            *   **报错**：`WSL: 检测到 localhost 代理配置，但未镜像到 WSL...`
            *   **认知**：**“WSL2 ≠ Windows”**。
                *   *细节*：你痛苦地发现，WSL2 里的 `127.0.0.1` 只属于它自己，根本够不到 Windows 的 localhost。你被迫放弃了最自然的直觉。

            #### 2. 第二阶段：暴力的迂回（歧途）
            *   **目标**：绕过 localhost 限制，哪怕不优雅。
            *   **尝试**：**`netsh portproxy`（端口转发）** + **`172.x.x.x` 网关 IP**。
                *   你在 Windows 管理员权限下执行了 `netsh interface portproxy add...`，试图搭建一座“网桥”。
                *   你在 WSL 里写死网关 IP：`export http_proxy=http://xxxxxxxxxx:xxxx`。
            *   **卡点**：**`Trying xxx.xxx.xx.xxx:xxx` (卡住，无响应)**。
            *   **死循环排查**：
                1.  你检查了代理软件，发现 `127.0.0.1:xxxx` 确实在监听（以为软件没问题）。
                2.  你检查了 `netsh` 规则，发现确实在监听（以为转发没问题）。
                3.  **真相**：Windows 的 **NAT 防火墙**在作祟。即使你搭建了桥，Windows 的底层防火墙依然拦截了来自 WSL 虚拟网卡（172.x.x.x）的“外部访问”，导致 TCP 握手直接被丢弃，造成了诡异的“卡死”。

            #### 3. 第三阶段：安全的执念（僵局）
            *   **痛点**：你拒绝开启代理软件的 **“Allow LAN”**（允许局域网）。
            *   **理由**：**“公共 WiFi 安全焦虑”**。
                *   *细节*：你担心开启 LAN 会让隔壁咖啡厅的人扫描到你的代理端口。这导致你死磕 `portproxy`，试图做一个“只对自己开放”的转发，结果陷入了 Windows 网络栈的深坑。

            #### 4. 终极顿悟：模式切换（破局）
            *   **转折**：发现 **Mirrored（镜像）模式**。
            *   **核心洞察**：之前的痛苦都是在 **NAT 模式**下产生的。NAT 模式把 WSL 当成了一台独立的物理机，需要走复杂的网络路由和防火墙。
            *   **解决**：在 wsl的GUI的网络设置中开启 `networkingMode=mirrored.，然后同时关闭localhost代理。'
            *   **结果**：WSL 和 Windows 变成了“同一个身体里的两个灵魂”，共享网络命名空间。
                *   你终于可以**重新使用 `127.0.0.1`**。
                *   你**不需要开启 Allow LAN**（因为这不再是局域网访问，而是本机进程通信）。
                *   你彻底摆脱了 `172.x.x.x` 的 IP 变化噩梦。
